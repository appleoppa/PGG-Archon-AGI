## 核心原理

DGM (Darwin Godel Machine) 和 HyperAgents 的研究揭示了一个关键事实：

> **单次深度分析的上下文容量是有限的。试图在单一上下文中做到"足够深"，会导致分析停留在表层。"不够深"不是能力问题，是方法问题。**

解决方案不是"再多读一遍"，而是 **并行分解-合成** 范式：
1. 将复杂分析任务分解为多个正交的子分析
2. 每个子分析在独立上下文中全力深入一个方向
3. 合成阶段合并各子分析的发现
4. 存档分析路径，供以后复用

## 核心原理补充（C18R2更新）

### 分阶段评估（staged evaluation，来自DGM-H/evolve）

不是所有子任务都需要全量深度分析。引入先筛后精模式：
1. **快速筛选阶段（screening）**：每个子任务先做1轮轻量扫描（只有单次调用，输出维度：发现/未发现/需深挖）
2. **深度挖掘阶段（deep dive）**：仅对筛选中标记"需深挖"的子任务做full-depth分析
3. 这样在不损失覆盖率的前提下，大幅降低token消耗

### 存档驱动（archive-driven，来自lemoz/darwin-godel-machine）

每次multi-pass执行后，将一个JSON摘要写入archive目录：
```json
{
  "task": "任务描述",
  "decomposition": ["子任务A", "子任务B", "子任务C"],
  "decomposition_quality": {"正交性": 0.8, "覆盖率": 0.7, "效率": 0.6},
  "synthesis_quality": 0.75,
  "missed_directions": ["未覆盖到的分解方向"],
  "improvement_suggestions": ["下次如何改进分解"]
}
```
这样下次类似任务时可以直接复用/改进分解策略，而不是从零设计。

## 工作流

### 第1步：分解识别

将复杂分析任务分解为独立的子分析任务。每个子任务应该：
- 聚焦一个特定的分析维度（如：公式结构、版本历史、文件间一致性、配置逻辑）
- 目标单一、不可再分
- 子任务间尽可能正交，减少重复

**好的分解示例**（来自C18审错实战）：
```
子任务A: 公式数学结构审计（寻找运算bug/权重异常）
子任务B: 版本历史一致性审计（跨文件的版本声明矛盾）
子任务C: 配置逻辑审计（硬编码值/参数传递问题）
子任务D: 实现一致性审计（相同概念在不同文件中的表达差异）
```

**文档审计分解示例**（来自C20边界清扫实战）：
```
子任务A: 路径一致性审计 — 扫描文档引用的所有文件/目录路径是否实际存在于文件系统。
           含两类：①路径偏移（文档说A实际在B）②声明式phantom（索引表引用从未创建的目录）
子任务B: 跨文件引用完整性审计 — 流程节点编号、章节引用、版本声明在各段落间的一致性
子任务C: 版本残留/内容过时审计 — 旧版本标签、已废弃的引用、过时的说明文字
```
此分解的正交性约0.85，覆盖率约0.9。路径一致性审计是C20R5新发现的维度——确保文档索引表的"应存在"等于文件系统的"实际存在"。

### 第2步：并行深度分析

使用 `delegate_task` 并行派发所有子分析任务。每个子任务：
- 有自己独立的上下文，专注于单一维度
- 被明确要求：深入阅读源码，不只读表面描述
- 要求输出"确认性结论"（不仅是发现，还包括"我确认这个部分没有问题"）
- 标记每条发现的置信度（高/中/低）

**关键参数**：
```python
toolsets = ['terminal', 'file', 'web']  # 根据任务需要选择
```

### 第3步：合成

将并行子分析的结论汇聚到主上下文。合成规则：
1. 所有子分析的结论都是平等的输入
2. 寻找子分析间的矛盾点（这是最有价值的部分）
3. 对于置信度<0.7的发现，在合成中标记为"待验证"
4. 确认每个子分析是否有盲区（该方向但未发现的）

### 第4步：存档与学习

每次执行完成后：
1. 记录本次分解策略（分了几路、各聚焦什么方向）
2. 记录是否错过了任何问题（事后看哪些分解方向没覆盖到）
3. 下次类似任务时加载上次的分解策略作为起点

## 应用场景

### 场景A：公式/代码审计（如C18审错）

传统方式：单上下文中反复读同一批文件3遍才发现不同的bug。
多轮次方式：4个子任务并行各深入一个方向：
- 数学结构子任务 → 发现运算bug和权重异常
- 版本历史子任务 → 发现版本矛盾（死维度）
- 配置逻辑子任务 → 发现硬编码值
- 实现一致性子任务 → 发现跨文件公式冲突

### 场景B：跨周期边界审计（C21实战）

在进化周期的 `cycle_summary` 阶段，需要审计本轮改动是否引入了新的不一致。

**分解策略**（C21边界清零实战 — 审计AGENTS.md在C21的4处修改）：
```
子任务A: 路径一致性审计 — C21R1新增的子任务输出路径规范在全文正确引用？
子任务B: 步骤编号一致性审计 — C21R3 Step⑦展开为4子步骤后，流程图/后续引用是否一致？
子任务C: ⚡命令准确性审计 — 所有新增的⚡命令参数是否与脚本实际定义匹配？
子任务D: 交叉引用完整性审计 — 13_feishu_delivery_standard.md是否与Step⑦对齐？
```

**结果**（2026-05-17 C21 boundary）：
- 发现6个问题：P0×1（`--id`参数不存在，命令必定失败）、P1×3（跨文件对齐偏差）、P2×2（风格不一致）
- P0是在本周期R3引入的——验证了边界审计能捕获round级遗漏的引入缺陷
- 修复后进行下一周期

**与场景A的关键区别**：
| 方面 | 场景A | 场景B（边界审计） |
|:----:|:-------:|:-----------------:|
| 审计范围 | 整个代码库/文件 | 仅本周期修改过的区域 |
| 目标 | 发现所有问题 | 发现引入缺陷 + 交叉引用缺失 |
| 分解依据 | 系统模块 | 本周期改动点×一致性维度 |
| 子任务数量 | 通常3-4路并行 | 通常1-2路（焦点集中） |

**单子任务模式**：当审计范围聚焦于单个文件的特定改动（如C21 boundary只审计AGENTS.md的4处C21新增内容），不需要并行分解。`delegate_task` 单子任务即可完成全部审计，前提是：
- 审计范围描述精确（文件、行号、改动内容）
- 明确要求交叉验证脚本与文档的一致性
- 要求按优先级输出（P0/P1/P2）

**C22边界审计实战（2026-05-17）** — 审计C22新增脚本命令与跨文件引用一致性：

**分解策略**（2路并行，正交性0.9，覆盖率0.95）：
```
子任务A: ⚡命令准确性审计 — C22新增self_check_scorer/audit_scanner/knowledge_precipitator命令参数与脚本实际CLI定义是否匹配
子任务B: 跨文件引用一致性审计 — Step⑥/⑧/⑤b/F节在AGENTS.md/办案流程详解.md/13_feishu_delivery_standard.md之间的一致性
```

**结果**：
- 8个发现：P0=0, P1=3, P2=5
- Sub A 确认所有⚡命令参数存在且退出码约定匹配文档
- Sub B 发现3处P1（办案详解.md Step⑧缺gate命令、13标准 Step⑧缺status+report、办案详解§5.11缺scorer引用）
- 全部3处P1当场修复

**与C21边界审计的关键区别**：
| 方面 | C21边界审计 | C22边界审计 |
|:----:|:-----------|:-----------|
| 周期焦点 | 流程效率（自动交付链+路径标准化） | 质量强化（自检+审计+知识沉淀） |
| 审计目标 | 新流程步的引用一致性和命令准确性 | 新创建脚本CLI与文档的匹配 + 跨文件引用 |
| 子任务A | 路径一致性（文档存在性） | 命令参数准确性（CLI vs 文档） |
| 子任务B | 步骤编号一致性（4子步骤扩散） | 跨文件引用一致性（3文件集） |
| 发现分布 | P0×1, P1×3, P2×2 | P0×0, P1×3, P2×5 |
| 结论 | 命令参数错误（`--id`不存在=P0必败） | 引用缺失（P1），无必败问题 |

**跨文件审计检测模式对比**：当审计范围跨越3个以上文件时，以下两种检测模式互补使用：
1. **search_files 单扫描**（通用）：用一个pattern跨目录搜索，适合检查某项引用/编号/路径在何处存在——但无法检测"该出现而未出现"的缺失
2. **逐文件章节对比**（精准）：对每个文件中的对应章节做逐行diff，适合发现"某文件缺少某内容"的P1/P2级缺失

## 实证案例：C18R3 AGENTS.md 审计（2026-05-17）

这是 multi-pass-deep-analysis 从创建到实战验证的完整案例。

### 任务
分析 ~/.hermes/workspace/AGENTS.md（1539行，67KB）的结构一致性、版本连贯性、角色边界。

### 分解策略
```
子任务A: 规则结构一致性审计（红线冲突/门控缺失/自相矛盾）
子任务B: 版本演化连贯性审计（版本关系/旧版本残留/跨版本协调性）
子任务C: 角色职责边界审计（中枢vs管理中心/巡视vs复核/轻桥层vs全流程）
```

### 结果
- 3路并行, 全在80-120秒内完成
- 总共发现 15 个问题
- P0级: "医生"残留2处, 保鲜机制流程缺失
- 存根: multi-pass-archive.json 存档

### 关键教训
1. **分解正交性是核心**: A/B/C三方向正交度高(0.9), 无重叠发现, 覆盖率达85%
2. **合成阶段的价值**: 子任务B(C版本审计)和子任务C(角色审计)都发现了"医生残留"——矛盾确认法验证了发现的可靠性
3. **archive驱动有效**: 存档的decomposition_quality字段记录了正交性0.9/覆盖率0.85/效率0.75，下次类似任务可基准对比

## 后处理：多层审查-修正循环（multi-pass的"验证修复"阶段）

multi-pass分析发现问题后，需要经过多层质量门控逐一验证修复。详见 `references/multi-layer-review-fix-cycle.md`。

核心模式：分解为多个正交审查维度（巡视组/律法支持部/业务主办部门内部格式自检），每轮使用Stage-Gate结论 + 必须修正项清单，逐轮消化。修正项分级为 M（必须修）和 S（建议修），并追踪版本链。

## 陷阱与注意事项

- **不是所有的任务都适合分解**：简单任务（查找单个函数bug）直接分析即可
- **子任务数量上限**：用户配置的最大并发数（当前：3）
- **合成阶段不要抛弃矛盾结论**：矛盾点通常指向最根本的问题
- **存档不要过度**：存档分析策略，不是存档分析结果（结果在session里）
- **分解方向需正交**：方向重叠浪费并行容量
- **第一次分解可能不完美**：事后复盘分解策略，下次改进——这就是Ξ提升
- **连续同标点→滞胀陷阱**：如果连续3+轮选择同一对标点且ΔG未实质性变化，说明该方向已到天花板，需要换标点或改变策略
- **创建≠能力**：创建skill不等于能力提升。只有实战验证（至少完成一次真实使用）才能确认skill有效。否则Ξ评分存在虚增风险。

## 存档JSON参考格式

每次 multi-pass 执行后写入 archive 的 JSON 格式（参考 multi-pass-archive.json）：

```json
{
  "task": "任务描述",
  "timestamp": "ISO时间戳",
  "decomposition": ["子任务A: 聚焦方向", "子任务B: 聚焦方向", "子任务C: 聚焦方向"],
  "decomposition_quality": {"正交性": 0-1, "覆盖率": 0-1, "效率": 0-1},
  "synthesis_quality": 0-1,
  "total_discoveries": N,
  "high_priority": N,
  "missed_directions": ["未覆盖的分解方向"],
  "improvement_suggestions": ["下次改进建议"]
}
```

## 自检清单（每轮multi-pass执行后）

- 正交性是否≥0.7？（≤0.5说明方向重叠，需重新分解）
- 覆盖率是否≥0.75？（≤0.6说明有盲区）
- 存档JSON是否已写入？
- 如果连续3次用同一分解模式，是否需要调整？

## Ξ维度提升量化

| 应用前 | 应用后 |
|:-----:|:------:|
| 3遍才能挖到根因 | 1轮并行+1轮合成 = 覆盖全面 |
| 跨文件推理盲区 | 专门的版本历史子任务保障 |
| 单次分析深度受限于上下文 | 每个子任务充分利用独立上下文 |
| 无复用模式 | 存档分解策略，下次直接加载 |
