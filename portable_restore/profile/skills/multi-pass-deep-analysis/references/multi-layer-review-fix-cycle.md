# 多层审查-修正循环（Review-Fix Cycle Pattern）

> 来源：016号民邦人力案办案全流程（V6.0）实战
> 时间：2026-05-17
> 场景：multi-pass审计发现问题后，通过多层质量门控逐一修复

## 问题背景

multi-pass分析发现的15项AGENTS.md问题修复后，需要验证这些修复是否真的生效、是否引入了新问题。但直接"修复→验证"的单一循环深度不足——同一上下文中难以同时做到 ①审计细节追溯 ②修正质量验证 ③边缘情况检查。

## 解决方案：多层审查-修正循环

不是单次"修复→验讫"，而是分解为多个正交审查维度，每轮审查独立聚焦一个方面，各产生"条件通过"结论+必须修正项，逐轮消化。

### 第一层：执行过程效验（巡视组）

聚焦：法条引用准确性 + 论证逻辑完整性 + 遗漏检查

**典型输出**：
- 法条引用效验：逐条核对，标注准确性/版本/适用性
- 论证逻辑效验：检查逻辑跳跃、未充分论证环节、分支充分性
- 遗漏检查：未覆盖的法律问题、风险因素、证据类型

**产出**：M1-M4必须修正项 + S1-S10建议修正项

### 第二层：依据复核（律法支持部）

聚焦：条文效验（条文编号+内容+有效性） + 论证一致性（引用与结论匹配度）

**典型输出**：
- 一级复核：逐条法条核查表（编号/内容/有效性/适用性）
- 二级复核：检查引用错误、引用遗漏、路径间法条协调性
- 与第一层交叉验证：确认第一轮的修正到位

**产出**：M5-M8必须修正项 + S11-S14建议修正项

### 第三层：格式校对（文书机要部）

聚焦：格式一致性（标题/表格/列表/引用/标点/术语/法条引用格式）

**产出**：格式修正列表 + 校对后完整内容

## 关键模式

### 1. "条件通过"优于"不通过"

每层审查结论使用 **条件通过**（Conditional Pass）而非直接不通过。好处：
- 不阻塞下游流程（文档可继续流转）
- 明确标注必须修正项（M1, M2, …）
- 给上游明确的修正方向，减少来回往返

### 2. 修正项分级

| 级别 | 含义 | 示例 |
|:----:|:------|:------|
| M (Must Fix) | 必须修正，否则下一层不接收 | M1: 缺失格式条款论证环节 |
| S (Should Fix) | 建议修正，不影响流程推进 | S6: 版本标注缺失 |
| 高/中/低 | 严重性标注 | 🔴🟡🟢 |

### 3. 版本链追踪

每次审查+修正产生一个新版本，在文档末尾记录：
```
| V1.0 | 初稿完成 |
| V1.1 | 巡视组节点①修正（M1-M4） |
| V1.2 | 依据复核修正（M5-M8） |
| V1.2签认 | 苹果中枢确认进入下一环节 |
```

### 4. 修正后更新版本号

每一轮修正后，必须：
- 更新文档中的版本号（正文 + 报告信息表）
- 在版本链表中追加新的版本行
- 确认旧版本的问题标记已清除

## 与multi-pass-deep-analysis的关系

```
multi-pass分析
   ↓  发现 N 个问题
多层审查-修正循环 ← 这是 multi-pass 的"后处理"阶段
   ↓  逐轮消化 M1...M8
格式校对定稿
   ↓
归档交付
```

multi-pass负责"发现问题"，多层审查-修正循环负责"验证修复"。两者构成了完整的 `分析→修复→验证→交付` 链条。

## 适用条件

- 审计/分析发现的问题超过5项时
- 问题涉及多个维度（法条/逻辑/格式/术语）时
- 需要多个部门/角色协同修正时
- 有严格的质量门控要求时

## 陷阱

- 不要在单次审查中试图覆盖所有维度 → 分解正交审查
- 不要跳过交叉验证（第二层确认第一层的修正） → 修正可能不到位
- 不要在修正时不更新版本号 → 版本链断裂导致溯源困难
- 不要在不通过后直接否决流程 → 改用条件通过+修正项清单
