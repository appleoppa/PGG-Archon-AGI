# 多模型路由与密钥安全配置经验

适用场景：用户把多个模型供应商的 API 密钥交给 Hermes，目标是让 Hermes 配置本机多模型路由，而不是让助手反复做安全说教。

## 核心原则

- 用户明确给密钥用于配置时，应主动配置，不要停在“建议轮换/建议安全存放”的解释上。
- 密钥可以写入本机密钥文件或环境变量，但不要写入：记忆、技能正文、周期报告、对外回复、路由状态输出。
- 配置文件优先只保留 `key_env` 引用；如果发现配置内已有明文密钥，应迁移到本机 `.env` 后删除配置内明文字段。
- 状态检查器只能输出“是否已配置/是否可用/角色/模型/传输模式”，不能输出密钥值、前后缀或可还原片段。
- 用户问“为什么换密钥”时，可以解释“聊天中明文出现即按暴露风险处理”，但如果用户明确说“我给你就是让你配置”，下一步应执行配置。

## 推荐配置形态

- 主模型仍写在 `model.default` 与 `model.provider`。
- 多模型路由作为非密钥配置保存，只记录角色与 provider/model 映射。
- provider 条目使用：`base_url`、`key_env`、`api_mode`、`model`、`models`。
- 密钥放在 `.env` 对应变量中，文件权限尽量设为仅用户可读写。

## 河图洛书式角色分配模板

- 中宫主脑：负责最终裁决、总判断、EVM 公式代入后的收束。
- 反证审错：负责拆错、反证、找漏洞、打断自嗨。
- 修复执行：负责快速提出修复路径、补短方案、落地草案。
- 旁证压缩：负责中文压缩、遗漏检查、轻量旁证。

### 调整既有超级路由时的落地规则

当用户明确调整多模型角色（例如“MiniMax负责主要调度，GPT作为最强主脑，DeepSeek作为辅助脑，进化以GPT为主”）时，不要只口头确认。应同时完成三类落地：
1. 更新主状态文件中的角色字段：主脑、主要调度、进化主导、辅助脑、决策顺序。
2. 更新相关说明/任务模板/门禁脚本里的角色链路，避免旧模板继续把模型写成旧顺序。
3. 新增或更新一条路由策略基因记录，写明当前有效策略和决策顺序。

推荐决策顺序可表达为：主要调度模型先拆分任务与检查点 → GPT主脑作核心判断 → DeepSeek辅助反证 → 调度模型复核证据链 → GPT最终收束。验证时只输出角色与模型是否对齐、脚本是否可运行；不要输出密钥、配置片段或底层命令。

## 执行顺序

1. 先读取 Hermes 配置规则或相关技能，确认当前版本支持的 provider/custom provider 写法。
2. 检查当前配置中是否已有主模型和现有密钥引用。
3. 若配置内有明文密钥，迁移到本机密钥文件，并把 provider 改为 `key_env` 引用。
4. 增加多个 provider 条目，分别设置端点、模型、传输模式和密钥环境变量名。
5. 增加非密钥路由配置，记录模型角色分工。
6. 编写或运行只输出状态的验证器，确认端点已配置、密钥是否存在、路由角色是否完整。
7. 用户侧汇报只说“已配置/可用/缺哪个密钥槽”，不要展示路径、命令、密钥、英文字段。

## 常见坑

- 不要把“安全提醒”误当成“拒绝配置”。用户明确授权配置时，正确动作是安全配置并不泄露。
- 不要把密钥写入长期记忆或技能文件；技能只能记录方法和变量名模板。
- 不要在飞书回复中展示原始工具输出、配置片段或英文状态字段。
- 如果某个模型密钥槽为空，只说明“接口已配置，密钥槽为空”，不要把它当成配置失败。
