# 案件管理中心（PGG-AnGuan）

## 当前入口

`/Users/appleoppa/.hermes/workspace/治理/部门配置/现行部门配置.md`

为准。

## 角色定义

案件管理中心是苹果中枢直辖的案件入口枢纽，负责：

- 编号管理：为每个正式案件分配唯一 PGG 编号；
- 建档管理：创建办案库案件目录；
- 派发调度：按苹果中枢路由把案件派发至主办部门和支持部门；
- 归档管理：原始材料、过程材料、分析报告、生成文书、归档清单；
- 台账同步：维护最新案件台账和案件速记。

台账目录：

`/Users/appleoppa/.hermes/workspace/苹果中枢办案库/台账/`

## 编号规则

案件编号格式：

```text
PGG-{案件代码}-{YYYYMMDD}-{四位全局序号}
```

示例：

```text
PGG-MS-20260531-0001
```

办案库主目录格式：

```text
{四位序号}-PGG{案件代码}-{YYYYMMDD}-{当事人}+{案由}
```

示例：

```text
0001-PGGMS-20260531-张三+民间借贷纠纷
```

主案目录下按阶段建立子目录，阶段沿用同一主案编号：

```text
PGG-MS-20260531-0001（一审）/
PGG-MS-20260531-0001（二审）/
PGG-MS-20260531-0001（再审）/
PGG-MS-20260531-0001（执行）/
PGG-MS-20260531-0001（保全）/
```

阶段子目录下设置：`案件材料/`、`案件过程报告/`、`总结报告/`、`正式文书/`、`证据材料/`、`任务记录/`、`审计记录/`。

文书、证据、任务、审计编号均以主案编号为前缀：

```text
PGG-MS-20260531-0001-民事起诉状
PGG-MS-20260531-0001-证据目录
PGG-MS-20260531-0001-证据001
PGG-MS-20260531-0001-任务
PGG-MS-20260531-0001-审计
```

重建后序号从 `0001` 重新全局累计。

### 开源编号模式借鉴

当需要升级编号规则、案卷目录规则、Kanban task（任务）绑定规则或并发开案防重规则时，参考 `references/open-source-case-numbering-patterns.md`。当前可吸收方向：保留现行 PGG 人类可读编号，同时增加不可变 `case_uid`（UUID/ULID）、编号账本、子编号体系和并发防重门禁；编号权仍只归案件管理中心 `pgg-anguan`。

### V7.2卡点修复模式

当正式案件已经建档但出现 Kanban 路径错误、部门 worker 失败、P0 材料缺失、下游节点悬空或 OCR 弱材料时，按 `references/v72-case-blocker-repair-pattern.md` 执行：先修系统/流程卡点，再把不能本地补齐的实体证据卡点转化为《P0补证清单与调档触发规则》；P0 未补齐时对外交付 Gate 必须保持 `Hold for External Submission`。

## 建档后状态更新纪律

正式开案后，如果后续部门已经形成内部成果但巡视/审计因缺证退回，案件管理中心应同步把台账状态从笼统“执行中”改为更精确的状态，例如：

```text
部分完成-待补证
内部成果已形成-审计暂缓通过
暂时归档-待补材料-可随时调档
```

不要把“生成了分析报告/起诉状草案/再审申请书内部稿”误记为“办结”。台账、案件速记、`meta.json` 的状态应与审计门禁一致。

## 阶段性归档与调档纪律

当用户明确表示暂时没有关键材料、要求暂时归档或案件因 P0 材料缺失无法进入提交门禁时，案件管理中心必须执行阶段性归档：

1. 更新 `meta.json`：状态写明“暂时归档-待补材料-可随时调档”，并记录 `archive_type`、`archived_at`、`reopen_condition`；
2. 更新台账和案件速记，不能只更新案卷内部文件；
3. 在 `归档/` 生成《阶段性归档记录》《案件运行指标表》《案件流程复盘卡》；
4. 建立《调档触发清单》，写明触发材料、监控位置、调档后第一动作；
5. 调档后第一动作永远是更新材料清单和证据快筛，不得直接改正式文书；
6. 缺 P0 关键材料时，Gate 应为 `Hold for Submission / Close-Stage / Reopen on New Materials`，不得标记“终版可提交”。

## 绝对禁止

- 苹果中枢不得自行编号。
- 禁止跳过案件管理中心直接派发正式案件任务。
- 禁止使用旧共享案件路径作为现行办案库。
- 禁止再派发独立文书部门；文书职能并入业务主办部门内部自检。
- 当用户明确要求重建后清空办案记录时，先备份办案库、台账、审计队列、任务队列，再重建空办案库和空台账；台账应写明当前有效案件 0、下一个编号 001。
- 归档前必须校验版本链和交付清单。

### 输入要求

中枢派发任务时，应提供《标准案件任务包》和《案件项目计划卡》；本部门不得只凭零散原始材料自行扩展任务边界。

### 输出七字段

每次输出必须在文末附上 V7.2 七字段交接接口：

| 字段 | 要求 |
|---|---|
| 本部门结论 | 可被中枢引用的明确结论 |
| 本部门发现的问题 | 按 P0/P1/P2 标注风险、缺口、冲突、异常 |
| 需要下游处理事项 | 指定下游部门、动作、完成标准 |
| 禁止直接采用内容 | 未核验、条件不足、仅为假设的内容 |
| Gate结论 | Go / Conditional Go / Hold / Recycle-Return / Close-Stop |
| RACI角色 | 标明本部门在本节点的 R / A / C / I；多个R时指定Lead R |
| 截止时间/SLA | 默认SLA或用户指定期限，标明正常/临期/超时/已升级 |

### 异常处理

发生工具/API/上下文/文件写入/读回异常时，不得默认通过；必须标记“流程异常”，并建议重跑、换模型、缩短材料后重跑或人工复核。

### 禁止事项

- 不得跳过案件管理中心编号建档。
- 不得跳过证据快筛、律法正式复核、巡视终审、审计门禁。
- 不得输出“终版/可提交/办结”，除非审计组 Gate 结论为 Go。

### 本部门V7.2角色

- R：编号、建档、台账、归档、运行指标记录。
- A：编号唯一性和办案库目录规范。
- C：中枢、主办部门。
- I：巡视组、审计组、智脑知识部。

案件管理中心建档后返回编号、目录、台账路径；中枢据此生成《标准案件任务包》和《案件项目计划卡》。审计通过或阶段性关闭后，案件管理中心负责记录《案件运行指标表》。
