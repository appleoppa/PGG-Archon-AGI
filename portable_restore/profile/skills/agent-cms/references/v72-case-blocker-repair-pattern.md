# V7.2办案卡点修复模式：路径、门禁、补证与Kanban状态

## 触发场景

正式案件已经建档并创建 Kanban task graph（任务图），但出现以下任一问题：

- task body 的 `input_paths` / `output_paths` 指向旧式 `办案库/{case_id}`，而真实案卷目录是现行 `{四位序号}-PGG{案件代码}-{YYYYMMDD}-{当事人}+{案由}`；
- P0 证据材料缺失，但正式文书、正式复核、巡视终审、审计、归档、交付节点仍为 `todo/ready`；
- 部门 worker 因认证或运行问题未形成真实 run evidence（运行证据），但中枢已经基于材料形成内部草案；
- OCR弱材料无法可靠提取文字，需要阻断正式引用。

## 修复原则

1. 先修系统/流程卡点，再处理实体证据卡点。
2. 能本地修复的立即修复：路径、脚本参数、Kanban状态、台账、meta、材料清单、OCR索引。
3. 不能本地补齐的实体证据不得伪造：工伤认定、鉴定结论、票据、主体关系、前案材料等只能转化为补证清单。
4. P0 未补齐时，对外交付 Gate 必须是 `Hold for External Submission`。

## 推荐动作

### 1. 案卷路径修复

- 创建 task graph 时必须把真实主案目录传入脚本；脚本应支持 `--case-dir`。
- 已创建的 task 如路径错误，应批量替换 task body 中旧路径。
- 读回至少一个下游任务，确认 `input_paths` 和 `output_paths` 指向真实案卷目录。

### 2. Kanban状态修复

当 P0 缺口存在时，下游正式节点不要停留在 `todo/ready`，应阻断：

- 主办二修 + 格式自检；
- 律法正式复核；
- 巡视轻量预审/终审；
- 交付前整合；
- 审计最终门禁；
- 归档沉淀；
- 中枢交付。

状态口径：

```text
内部分析：Conditional Go
对外交付：Hold
重启条件：Reopen on New Materials
```

### 3. 补证清单

必须生成《P0补证清单与调档触发规则》，至少包括：

- 关键实体材料；
- 当前缺口；
- 收到补充材料后的入库路径；
- 更新材料清单、OCR、证据目录、门禁复核的顺序；
- 禁止直接生成/修改正式文书的提醒。

### 4. 修复记录

生成《卡点修复记录》，区分：

- 已修复的系统卡点；
- 已修复的流程卡点；
- 仍需当事人/外部材料补充的实体卡点；
- 读回验证证据。

## 完成态门禁

交付前必须读回验证：

- 关键文件存在且大小非零；
- meta 状态与台账状态一致；
- Kanban 统计不存在误触发的 `ready/running` 正式交付节点；
- task body 路径指向真实案卷目录；
- 审计记录明确 `Hold` 或 `Go`，不得含糊。
