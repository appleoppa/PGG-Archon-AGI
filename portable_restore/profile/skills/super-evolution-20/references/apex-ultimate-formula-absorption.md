# APEX Ultimate formula absorption pattern

Session-derived reusable pattern for future SE20 / PGG Archon formula upgrades.

## Trigger

Use when the user provides a revised top-level evolution formula, axiom, governance baseline, or renaming of the AGI/APEX/PGG Archon formula stack.

## Durable workflow

1. Treat the user-supplied formula as the new semantic target, but keep engineering boundaries explicit: it becomes a governance/scoring/prompt-injection baseline, not a claim of third-party model weight control or full AGI.
2. Map every symbolic layer to an executable or auditable field before saying it is learned:
   - `Ω_A` → memory / akashic retrieval layer
   - `α_ack` → convergence / contradiction gate
   - `β_bg` → post-evaluation / growth loop
   - `ΨΛΓΞΦΥ` or `Θ_stable` → six deterministic kernel factors
   - `EVM` → entropy/frequency scoring
   - `Tao` / `A·B·TDHLGWB` → classical balance / behavior dimension layer
   - `ΣΔ_all` → defect deduction / fail-closed gate
3. Patch class-level implementation surfaces rather than creating a one-off report:
   - formula preamble / compact formula
   - bounded sidecar scorer/report builder
   - manifest generator and generated manifest
   - tests that assert formula string, compact string, six-factor layer, critical-delta fail-close, and read-only boundary
   - the governing skill itself
4. Run targeted tests plus readback verification before final reply. A useful smoke check prints schema, full formula, compact formula, score/status, six kernel factors, and Tao product.
5. Preserve compatibility names only as historical aliases. New user-visible names should follow the latest user-provided formula name.
6. Avoid staging or committing unrelated working-tree changes. Use file-specific status/diff checks when the repo already contains unrelated modified files.

## Pitfalls

- Do not answer only “learned” if code/manifest/skill surfaces can be safely updated.
- Do not claim global force-inheritance means overriding system/developer messages, hidden reasoning, credentials, or third-party model weights.
- Do not collapse a rich formula into a single scalar; retain symbolic layers in report fields so later gates can inspect them.
- Do not create a narrow skill named after the session formula; update the umbrella evolution/formula skill and put session detail here.
