# run_conversation RC characterization → guarded extraction workflow

## Trigger
Use this reference when a PGG Archon / Hermes evolution task touches `agent/conversation_loop.py::run_conversation`, especially when the function is large, high-branching, or tied to provider retry, tool execution, persistence, scheduler, or security-adjacent paths.

## Durable workflow learned
1. **Build read-only characterization gates before extraction.** Split the large function into named RC windows (for example RC-S01..RC-S09) and assert stable behavior signals with AST-backed contracts before moving code.
2. **Create an extraction readiness matrix.** It should explicitly state: all RC contracts PASS/WATCH, allowed extraction scope, forbidden actions, rollback requirements, and a conservative readiness score.
3. **Extract in scoped commits, but allow larger stage helpers when explicitly requested.** Default to one small mechanical helper per commit. If the user says “继续 / 步子大一点” and the risk score is >75 with rollback available, a larger stage-level helper is acceptable inside the current RC boundary. Prefer pure bootstrap helpers first: runtime reset, task id binding, runtime-main binding, input sanitization, turn message preparation, system prompt + preflight compression. Preserve no-behavior-change semantics and back every stage helper with behavior-level characterization tests.
4. **RC-S02 entry rule.** Do not jump straight into provider calls or tool execution. The first safe RC-S02 extraction target is pre-provider, pre-tool context collection such as `pre_llm_call` plugin context. Characterize dict/string context joining, copied `conversation_history`, `is_first_turn`, nonfatal hook errors, and “inject into user message, never system prompt” behavior before committing.
5. **Call GPT + Claude for co-review and MIMO as third-party auditor when the user asks for multi-LLM execution.** Store redacted evidence and hashes; never expose provider secrets from config.
6. **Keep window detection resilient to line drift and helper extraction.** Once helpers are inserted above or inside `run_conversation`, fixed line/window contracts can drift and falsely report WATCH. Anchor characterization windows relative to `run_conversation` and allow contract signals to be satisfied by either the local RC window or all-function/helper-call AST signals when the semantic code still exists.
7. **Verify after every helper extraction.** Minimum gate: targeted pytest, py_compile for edited files, `git diff --check`, APEX-GOD health with the repo venv, EVOLUTION_MANIFEST update, report, and a focused commit that stages only current files.
8. **Do not look stopped during long continuation loops.** If a validation/gate failure appears while the user has authorized continuation, briefly state the concrete blocker (for example “window drift causing WATCH”), then immediately keep patching/testing. Do not stop at a plan or ask whether to continue unless the next step crosses scheduler/security/provider/tool-execution boundaries.
9. **Honor the user's scored-continuation cap.** For this class of evolution/refactor task, when the next action scores >75%, is low-risk, and is rollback-safe, continue automatically without asking. If the user gives a max continuous-round cap (for example “≤5”), execute only up to that many fully verified/report/commit rounds, then stop or re-score. A candidate that is borderline (roughly 74–78 because it crosses alternate runtime/provider/tool/scheduler/security boundaries) should not be forced through; first add characterization tests to raise confidence, then re-score.
10. **Use a safe RC-S02 extraction ladder.** After RC-S01, prefer pre-provider/pre-tool helpers in this order before touching provider/tool execution: pre-LLM plugin context, memory-manager turn start + external memory prefetch, pre-loop interrupt/thread binding, pre-loop counter/file-mutation state initialization, and pre-turn stale connection cleanup. Treat alternate runtime early returns (for example codex app-server handoff) as boundary-adjacent: characterize before extracting.
11. **Provider-prep entry rule after pre-loop cleanup.** Before extracting provider request preparation, add a real-source AST contract covering `api_messages`, `api_kwargs`, retry state, response slot, sanitation/repair calls, prompt caching, token estimation, surrogate sanitation, and system-message injection. Start with retry/auth/recovery state initialization (`_initialize_api_retry_state`) rather than the provider call or API message builder. After extraction, update the contract anchor so `run_conversation` is allowed to call the helper while the helper’s own behavior test preserves exact defaults.

## Pitfalls
- Do not claim CodeGenesis PASS simply because gates passed; if high duplication remains, keep CodeGenesis as WATCH.
- Do not claim `run_conversation` is refactored after only helper extraction; state the exact helpers extracted.
- Do not touch RC-S02+ or scheduler/security boundary while the active decision only allows RC-S01 minimal extraction. If the user explicitly says to continue after RC-S01 stage helpers are complete, RC-S02 may start only with pre-provider/pre-tool context collection and fresh behavior tests.
- Do not preserve absolute line windows after adding helpers above the function; line drift can produce false WATCH/FAIL. When a stage helper moves semantic signals out of a fixed RC window, update the contract to check stable AST anchors/all-function signals instead of weakening the contract.
- Do not treat repeated progress pings as noise: when a user asks “怎么停了” during a long gate/fix loop, the workflow skill should force a short status + immediate next tool action pattern.
- Do not exceed a user-specified continuous execution cap. A cap like “小于等于5次” is a hard stop for automatic rounds; each counted round should include implementation, tests/gates, health/manifest, report, and scoped commit. Stop earlier if the next candidate does not stably exceed the >75 threshold.
- Do not mechanically extract alternate runtime early-return branches just because they are short. If a branch changes runtime handoff semantics (for example `codex_app_server`), first add behavior-level characterization tests, then re-score before extraction.

## Useful evidence fields
Report each iteration with: helper name, files changed, tests, health result, manifest generated_at + sha256, commit hash, LLM evidence hashes when used, and remaining boundary.
