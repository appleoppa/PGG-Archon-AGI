# 2026-05-23 超级进化12：评估中心与开源吞噬机制复现

## 触发背景

用户要求把“超级进化12-吞噬自进化”从方法论推进成真实能力，同时明确追问：

- 训练模型是否已完成；
- Hermes 原生核心改造是否已做；
- 自动吞噬全球开源前三项目是否已做；
- 继续做之前要先用“最强大脑”评估和审计。

## 关键用户偏好

- 高风险系统改造前必须先做评估与审计。
- 不接受把路线图、状态字段、脚本产物冒充真实完成。
- 要明确区分：已完成 / 部分完成 / 未做 / 只读采集 / 沙箱复现 / 核心主链路改造。
- 授权后可以继续推进，但应选择低风险、可回滚、安全边界清晰的路径。

## 有效推进模式

### 1. 先判定真实性状态

对每个能力点做证据分级：

| 状态 | 判定标准 |
|---|---|
| 已完成 | 有代码/配置入口、运行记录、产物、读回验证 |
| 部分完成 | 有 MVP 或安全接口层，但未进入主链路或未完全自动化 |
| 未做 | 只有路线图、报告、设想，没有真实执行产物 |
| 只读采集 | 只读取外部元数据/README/license，不克隆源码 |
| 沙箱复现 | 在隔离 workspace 中自写 mock/机制验证，不复制外部源码 |
| 核心改造 | 修改 Hermes 主链路源码并验证；风险最高，默认不优先 |

### 2. 安全接入顺序

推荐顺序：

```text
只读审计
→ 安全接口层 / sidecar / cron no-agent
→ 评估中心状态输出
→ workspace 沙箱机制复现
→ 固定评测集
→ 低风险能力接入
→ 最后才考虑核心主链路改造
```

禁止跳过许可证/边界直接克隆或复制外部项目代码。

### 3. 评估中心别名

用户已将“评估中心”定义为 Hermes 的轨迹评分、训练、预测和评估流水线。以后用户说：

- “打开评估中心”
- “评估中心怎么样了”
- “评估中心跑一轮”
- “评估中心为什么分低”

应默认理解为这套训练/评分/预测/评估体系，不要反复确认概念。

### 4. 外部开源吞噬的安全定义

“吞噬”分层：

1. **采集层**：GitHub 元数据、README、license，只读。
2. **能力抽取层**：总结机制、流程、架构，不复制表达性代码。
3. **沙箱复现层**：Hermes 自写 mock/最小机制验证。
4. **评测固化层**：把沙箱场景加入评估中心固定样例。
5. **能力接入层**：仅把验证过的机制接入 Hermes 安全接口或技能。
6. **核心改造层**：最后才考虑主链路源码改造，需要更严格备份、回滚、测试。

### 5. 本次形成的三个可复用机制

| 来源项目 | 抽取机制 | Hermes 复现方向 |
|---|---|---|
| obra/superpowers | 技能门禁开发闭环 | 需求澄清 → 计划 → TDD → Review → 完成前验证 |
| langchain-ai/langchain | Agent Graph + Trace + Evaluation | 计划节点 → 工具节点 → 观察节点 → 回答节点 → 评估节点 |
| TauricResearch/TradingAgents | 多角色正反审议 + 风险仲裁 | 支持方 → 反对方 → 风险官 → 最终裁判 → 决策日志 |

## 易错点

- 不要把“只读采集 3 个 GitHub 项目”说成“已自动吞噬全球前三项目”。
- 不要把“安全接口层/sidecar”说成“Hermes 原生核心主链路已改造”。
- 不要把“训练链路跑通”说成“模型效果可上线”。
- 不要把“沙箱机制复现”说成“完整复刻外部项目”。
- 不要直接修改 `config.yaml`、`.env`、`auth.json` 或 Hermes 主链路源码，除非用户明确授权并已准备回滚。
