# v0.6.0 Profile Gateway Autostart Analysis

## Discovery context
Discovered during a Web UI restart cycle where `hermes-web-ui stop` + `hermes-web-ui start` caused the server to hang for ~3 minutes. The `HERMES_WEB_UI_SKIP_PROFILE_GATEWAY_AUTOSTART` env var (documented for v0.5.x) was confirmed missing from the v0.6.0 compiled JS bundle.

## Evidence
- `grep -c 'SKIP_PROFILE' dist/server/index.js` returns 0 matches
- `grep 'process\.env\.[A-Z_]*' ... | sort -u` does NOT list `HERMES_WEB_UI_SKIP_PROFILE_GATEWAY_AUTOSTART`
- The autostart is hardcoded in the bootstrap sequence

## Symptom
- Node process starts (PID visible via `ps`)
- Port 8648 is NOT listening for ~3 minutes
- `logs/server.log` shows sequential `[gateway-autostart] gateway started via Hermes CLI service` + `[gateway-autostart] gateway start completed but did not report running within timeout` pairs for each profile
- Each profile cycle takes ~16 seconds
- Server eventually binds and serves after N × ~16s

## Timeline from real session
With 14 profiles (default + deepseekv4 + 12 PGG departments):
- Profile 1-2 (default, deepseekv4): ~0s each (already running, instant skip)
- Profiles 3-14 (PGG departments): ~16s each due to gateway start → wait for "running" → timeout
- Total: ~3 minutes
