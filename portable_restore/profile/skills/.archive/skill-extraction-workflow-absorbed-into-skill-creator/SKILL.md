# 技能萃取工作流

自动从过往会话中萃取可复用的知识模式，经去重和质量审查后，生成技能创建提案供用户审批。

## 核心理念

- 质量高于数量：不自动创建技能，只做发现+提议
- 周期性归纳（每周）优于即时创建：萃取真正的模式而非碎片记忆
- 生命周期闭环：扫描 → 比对去重 → 质量审查 → 提案 → 审批创建
- 每个新技能必须通过质量门槛

## 触发方式

- cron 任务自动触发（周一10:00，与现有定时任务对齐）
- 用户手动触发："检查技能萃取提案" / "提取可复用模式"

## 四阶段流程

### 阶段一：扫描（Scan）

目标：从过去7天的会话中识别值得沉淀为技能的模式。

搜索策略：
1. 用 session_search 搜索近期会话（过去7天）
2. 关注以下信号：
   - 用户纠正了我的行为（"不要这样"、"记住这个"）
   - 解决了一个复杂的技术问题（3+工具调用，有试错过程）
   - 发现了一个重复出现的工作模式（定期执行同一类操作）
   - 解锁了一个新的有用能力（发现自己能做以前不知道的事）
3. 对每个候选，提取：
   - 模式描述（这个模式解决什么问题）
   - 触发条件（什么情况下需要它）
   - 关键步骤（怎么做）
   - 坑和注意点（别人在这上面摔过什么）

输出：候选模式列表（每个包含名称雏形、描述、步骤要点、坑）

### 阶段二：比对去重（Dedup）

目标：每个候选与现有技能比对，避免创建重复技能。

比对维度（参考 skill-auto-maintain 的评分体系）：
1. 名称重叠：候选名与技能名中的分词交集
2. 关键词重叠：候选描述与技能描述/标签中的关键词交集
3. 结构重叠：候选的步骤/分类与现有技能的结构相似度
4. 关联引用：候选是否与现有技能引用链相关

决策规则：
- 与现有技能高度重叠（相似度 ≥ 0.50）→ 丢弃，记录原因
- 部分重叠（0.25 ≤ 相似度 < 0.50）→ 推荐合并到现有技能
- 无重叠（相似度 < 0.25）→ 进入下一阶段

输出：去重后的候选列表 + 合并建议

### 阶段三：质量审查（Vet）

目标：确保每个提案满足技能创建标准。

硬性检查项：
- [ ] description ≥ 20字符，非通用描述（不含"示例"、"默认"、"a skill for"等）
- [ ] 有明确的触发条件描述（"当用户要求..."）
- [ ] tags ≥ 2个（精准关键词，非泛词）
- [ ] 有可操作的步骤（不只有背景说明）
- [ ] 有常见坑/教训（至少1条）
- [ ] 整体内容 ≥ 300字符，非占位符

软性检查项：
- [ ] 这个模式是否在过去7天中出现 ≥ 2次？（复现验证）
- [ ] 解决的问题是否值得为一个独立技能？（必要性判断）
- [ ] 是否可以被现有某个技能吸收而非独立创建？（合并优先）

通过标准：硬性检查全部通过 + 软性检查至少 2/3 通过

输出：通过/否决标注 + 质量评分

### 阶段四：提案（Propose）

目标：生成用户可审批的提案报告。

提案格式：
```
---
## 技能萃取提案 [YYYY-MM-DD #N]

本次共发现 X 个候选模式，其中：
- 通过审查：Y 个
- 建议合并入现有技能：Z 个
- 重复/不合格：W 个

### 推荐创建

1. [技能名称]
   描述：...
   触发器：...
   核心步骤：...
   合并建议：无 / 建议合并入[技能名]
   质量评分：[PASS/NEEDS_REVIEW]

2. ...

### 推荐合并

1. [候选名] → 合并入[现有技能名]
   原因：...

### 已丢弃

1. [候选名] - 原因
```

## 质量创建标准（所有新技能必检）

创建新技能前必须满足：

```
前置检查：
[ ] 名称 ≤ 64字符，小写+连字符
[ ] description ≤ 1024字符，以"当用户..."开头
[ ] 有 frontmatter（name, description, version, author, license, metadata.tags）
[ ] 与已有技能不重复（执行 skill_view 或 search_files 验证）
[ ] tags ≥ 2个，精准非泛词
[ ] 含有可操作的步骤（非纯说明性文字）
[ ] 含有常见坑/教训（≥ 1条）
[ ] 整体内容 ≥ 300字符
[ ] 如果是扩展现有技能优于独立创建 → 使用 patch 更新而非新建
```

## 首次运行经验（2026-05-18）

首次扫描覆盖过去14天，20个会话产出63条信号，去重后7条。但7条全部是 R65 进化周期的 cron 模板日志和系统执行日志——没有一条是真正的"可复用工作模式"。

原因：扫描窗口内的会话以进化周期 cron 执行为主，没有真实的办案/开发会话。这证明了萃取系统依赖真实工作产出——没有真实的复杂任务，就产不出有价值的模式。

预期：随着正常办案开展，signal-to-noise 比例会自然改善。R65 进化周期的日志模式（格式固定的 cron prompt、模板化自诊输出、系统 exec completed 日志）应作为新老系统切换期的预期噪声源，在 LLM 策展步骤中主动过滤。

## 常见坑

1. 过度萃取：不是每个任务都值得存为技能。一个修正、一次搜索、一次文件操作都不构成技能。只有**可复用的模式**值得保存。
2. 重复创建：创建前必须先搜索现有技能确认不重复。宁可扩展已有技能也不新建高度相似的。
3. 描述空洞：description 必须让 agent 一眼判断"这个我需不需要加载"，"描述"、"工具"等泛词不合格。
4. 脱离触发条件：技能的核心价值在于自动触发。没有明确触发条件的技能大概率不会被用到。
5. 先问后做：创建技能前先向用户提案，用户批准后再执行 create。不要自作主张批量创建技能。

## 验证检查清单

- [ ] 提案已生成并存放在 workspace/技能萃取/ 目录下
- [ ] 每个提案候选都已执行去重比对
- [ ] 质量审查结果可追溯
- [ ] 用户审批后才执行 create
- [ ] 创建后更新 memory 记录该技能已存在防重复
