# 技能质量标准清单

每个新技能创建前的必检清单。可打印为 markdown checkbox 在创建前逐项核验。

## 前置检查

- [ ] 名称 ≤ 64 字符，小写+连字符（如 `skill-extraction-workflow`）
- [ ] description ≤ 1024 字符，以"当用户..."开头指明触发场景
- [ ] 完整 frontmatter：name, description, version, author, license, metadata.hermes.tags
- [ ] 与已有技能不重复（skill_view 或 search_files 验证）
- [ ] 可被现有技能吸收而非独立创建

## 内容质量

- [ ] tags ≥ 2 个，精准非泛词（不可用"工具"、"描述"、"技能"等）
- [ ] 含有可操作的步骤（至少 3 步，非纯说明性文字）
- [ ] 明确触发条件（"当用户要求..."格式）
- [ ] 含有常见坑/教训（≥ 1 条）
- [ ] 整体内容 ≥ 300 字符
- [ ] 非纯模板/占位符

## 格式规范

- [ ] 文件以 `---` 开头
- [ ] 第一章节为 `# 技能名`（H1）
- [ ] 有 H2 结构标题（如 ## 概述、## 触发条件、## 步骤等）
- [ ] 代码块用 ``` 包裹
- [ ] 路径引用使用绝对路径或相对 skill_dir 的路径

## 生命周期自检

- [ ] 这个技能在未来 3 个月内会被用到几次？（< 3 次 → 考虑不放技能）
- [ ] 是否可以用 memory 或 session 搜索代替？（单次教训 → memory 更合适）
- [ ] 这个模式是否已经以其他形式存在？（技能、文稿、笔记）

## 否决项（一票否决）

- description < 20 字符
- 包含"示例"、"默认"、"a skill for"等通用描述
- 无 trigger 定义
- 和现有技能重复率 ≥ 50%
- 文件内容 < 200 字符
