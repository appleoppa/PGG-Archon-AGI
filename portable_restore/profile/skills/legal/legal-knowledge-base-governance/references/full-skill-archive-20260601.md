# 法律知识库治理

## 何时使用

当用户要求重构、清理、扩展、导入、检索或评测法律知识库、向量库、案例库、法律法规库、司法机构库、RAG（检索增强生成）系统时使用。

## 核心原则

1. **本地官方库优先**：先盘点用户本地已存官方指导案例库、法律法规库，再考虑外部采集。
2. **外部公开源只补充**：法院、检察院、公安、司法行政等公开官网可以作为补缺与校验，不能覆盖本地官方主库。
3. **不只做向量库**：法律检索必须同时保留结构化条文、案例字段、编号精确检索、FTS/BM25、metadata filter（元数据过滤）、rerank（重排）、向量/图谱预留。
4. **编号精确命中高于语义相似**：案例号、法释号、条号、案号精确命中必须压过普通关键词或向量相似度。
5. **合法边界**：只采集公开资料或用户合法持有资料；不得获取非公开内部通讯录、私人联系方式、内网/政务专网/受限系统数据。

## 推荐流程

1. 先定位本地官方库：
   - 指导案例库
   - 法律法规库
   - 本地办案沉淀库
   - 司法机构/通讯录/位置类合法数据
2. 盘点文件：数量、路径、格式、hash、mtime、目录分类、来源层级。
3. 建立主表：
   - `local_official_assets`
   - `documents`
   - `legal_documents`
   - `guiding_case_fields`
   - `law_article_fields`
   - `law_articles_local`
   - `duplicate_resolution_v2`
   - `search_rank_rules`
   - `local_retrieval_eval`
4. 字段化：
   - 指导案例：案例号、标题、关键词、案由、争议焦点、裁判要旨、裁判规则、来源。
   - 法律法规：标题、发布机关、文号/法释号、发布日期、效力状态、类别、条文结构。
5. 条文级解析：按“法 / 编 / 章 / 节 / 条 / 款 / 项”逐级切分；至少先按 `第X条` 建结构化条文表。
6. 去重策略：content hash + 案例号 + 法释号 + 标题归一化；本地官方保留，外部重复项降级为 supplemental（补充来源），不要物理删除。
7. 检索排序：
   - 案例号精确命中：最高权重
   - 法释号/文号精确命中：最高权重
   - 条号精确命中：高权重
   - 本地官方库：加权
   - 外部补充源：降权
8. 建立 golden questions（黄金问题集）：覆盖案例号、法释号、条号、部门法关键词、办案常用问题；每轮改造后跑评测并读回结果。

## 用户偏好与坑点

- 用户明确纠正过：其本地已经存了大量官方指导案例库和法律法规库。此类任务如果不先查本地库，而先做外部公开网页样例采集，会被视为错误路线。
- 用户要求：可行性/收益/安全综合评分大于 75% 且低风险可回滚时，直接继续执行，不停在“建议下一步”。
- 汇报要简洁、字段化、有数量和证据文件；不要用漂亮报告掩盖没有接入本地官方库的事实。
