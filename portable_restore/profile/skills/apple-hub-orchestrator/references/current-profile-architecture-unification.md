# 当前 profile 架构统一与旧版清理门禁

适用场景：苹果哥说明对话丢失、上午/近期重新配置了 profile，要求把苹果中枢办案系统的架构、流程、标准统一，并在审核通过后清除旧版。

## 关键学习

1. **default 是现行苹果中枢**：当用户明确说明 `default` 对应旧系统 `main` 时，不要再把旧迁移文档中的 `main` 当作另一个待建 profile。
2. **pgg-* profiles 是现行部门**：以当前 Hermes profiles 为运行层证据；旧 `办案流程详解.md` 只是迁移时带来的历史流程底稿。
3. **标准办案流程可继续有效**：用户可能只要求统一“架构/部门/profile/标准”，不是推翻流程。不要把旧文档存在视为流程错误。
4. **先出待审核统一版，不直接清理旧版**：架构统一类任务必须先生成 `*_待审核.md` 或等价审阅稿，待苹果哥审核通过后再备份、改写、归档旧版。
5. **旧版清理要分阶段**：审核前不删除、不覆盖、不迁移；审核后也先备份、再同步入口/流程/模板/标准/skills，最后验证并归档旧版。

## 推荐执行顺序

```text
读取现行入口（若存在）
  ↓
核验 Hermes profiles：default + pgg-*
  ↓
读取旧流程/模板/标准，只作为待统一素材
  ↓
按用户最新口径生成现行架构待审核版
  ↓
生成旧版清理方案待审核版
  ↓
读回验证文件存在、关键口径、profile 数量
  ↓
汇报审核点，不擅自清理
```

## 待审核文件建议

- `workspace/现行部门配置_待审核.md`
- `workspace/旧版统一清理方案_待审核.md`

正式落位前不要覆盖 `workspace/治理/部门配置/现行部门配置.md`，除非用户明确审核通过。

## 决策点模板

| 决策点 | 必问/必标注 |
|---|---|
| default 是否就是苹果中枢/main | 是 |
| pgg-* profile 是否全部保留为部门 | 是 |
| 文书机要部是否正式部门 | 是，尤其旧文档可能写过下架 |
| 强制执行部是否恢复独立 profile | 若当前无 `pgg-zhixing`，必须列为待确认 |
| 旧模板/旧流程如何处理 | 审核后先备份再归档，不直接删除 |

## Pitfalls

- 不要凭旧迁移文档断言“文书机要部已下架”或“现行部门只有 V6.1 精简版”。当前 profile 架构可能已经被用户重新建立。
- 不要把“现行部门配置.md 缺失”解释成系统未建立；应从 profiles、SOUL.md、workflow、template、办案库台账恢复草案。
- 不要在未审核时清除旧版。用户要的是先统一、后审核、再清理。
