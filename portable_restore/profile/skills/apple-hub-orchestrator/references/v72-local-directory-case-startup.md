# V7.2 本地目录正式开案启动模式

适用场景：用户给出本地案件材料目录，并明确说“执行办案程序 / 启动办案 / 开案 / 开始办案”。

## 必做顺序

1. 加载办案入口与案管/证据/OCR相关技能。
2. 扫描源目录，统计有效文件，排除 `.DS_Store` 等系统噪音。
3. 由案件管理中心规则分配新案号，创建现行主目录：
   - `0003-PGGMS-YYYYMMDD-当事人+案由/`
   - `PGG-MS-YYYYMMDD-0003（一审）/案件材料/案件过程报告/总结报告/正式文书/证据材料/任务记录/审计记录/`
4. 将源材料复制入阶段目录的 `案件材料/原始材料/`，生成材料清单和 hash/大小记录。
5. 创建 V7.2 Kanban 任务图。注意：新规则下主目录不是 `CASE_ROOT/case_id`，必须向脚本/任务体传入真实 `case_dir`，否则 Kanban input/output path 会指向不存在的旧兼容目录。
6. 生成《标准案件任务包》和《案件项目计划卡》。
7. 对 PDF/图片执行文本提取：文本型 PDF 用 `pdftotext`，扫描 PDF/图片用 400DPI OCR；关键图片 OCR 结果必须标注“需人工对照原件”。
8. 生成证据快筛、内部初评、巡视/审计门禁。若 P0 证据缺失，Gate 只能是 `Conditional Go for Internal Analysis / Hold for External Submission`。
9. 更新 `meta.json`、`案件台账_最新.md`、`案件速记.md`。
10. 最终汇报必须列明：案号、案卷目录、文件数量、OCR结果、Kanban状态、Gate、不能对外交付的原因。

## Worker 真实性门禁

- 只有真实 Kanban worker run 成功并写入 output_files，才能说相应部门已正式执行。
- 如果 dispatch 或 profile provider 失败，可以由中枢基于真实材料生成“内部草案/中枢代拟”，但必须显式写明：不是正式部门 worker 完成，不得冒充多 profile 闭环。
- 这种情况下，后续正式文书、正式依据复核、巡视终审、审计 Go、对外交付都应 Hold，直到部门 worker 或人工复核补齐。

## 常见输出文件

- `任务记录/{case_id}-任务-编号建档.md`
- `任务记录/{case_id}-任务-材料清单.md`
- `任务记录/{case_id}-任务-标准任务包与项目计划卡.md`
- `案件材料/{case_id}-材料提取索引.json`
- `案件材料/{case_id}-材料提取全文合并.txt`
- `证据材料/{case_id}-证据目录.md`
- `案件过程报告/{case_id}-主办初评.md`
- `总结报告/{case_id}-内部初步分析报告.md`
- `审计记录/{case_id}-巡视轻量预审.md`
- `审计记录/{case_id}-审计最终门禁.md`

## 桌面同步纪律

除非用户明确要求“输出到桌面/发给我/同步桌面”，正式开案过程产物只进入办案库，不主动复制到 Desktop。
