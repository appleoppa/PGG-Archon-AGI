# 阶段性办案审计 blocked 后的处理模式

适用场景：正式办案流程已完成编号、部门分析、证据管理、推演、巡视、依据复核、知识沉淀，但交付前自检或审计未达标。

## 核心原则

1. **不得把阶段性成果包装为最终交付**
   - 自检未达 80% 或审计记录为 blocked 时，禁止使用“终版、定稿、交付版”等文件名。
   - 可使用“阶段性法律分析报告”“当前材料版”“补证指引版”。

2. **审计 blocked 不是失败终点，而是修复入口**
   - 写入审计记录后，必须把 blocked 原因转成可追踪任务。
   - 典型任务：补证清单、外部依据核验、类案核验、诉请金额校准、meta 状态更新。

3. **meta.json 必须同步真实状态**
   - 不能停留在“已编号建档/待派发”。
   - 应写明：阶段性办案完成、自检分数、审计结论、禁止外部交付、下一步动作。

4. **补证清单要结构化**
   - 不只写自然语言“需补充材料”。
   - 至少包含：证据名称、证明目的、优先级、当前状态、缺失风险。

5. **外部核验要任务化**
   - 凡报告中出现“需外部原文核验/需类案核验”，应生成独立核验清单。
   - 正式文书中出现“类案支持、法院普遍认为、裁判观点”等表达，必须绑定案号、法院、来源或明确标记不可引用。

6. **进化系统输入要来自审计短板，不来自漂亮总结**
   - 将审计报告中的短板提炼为进化输入。
   - 重点记录系统性问题：命名状态不一致、meta 滞后、补证未任务化、外部核验未闭环、类案硬门禁不足、自检失败后自动退回不完整。

## 推荐文件序列

在案件目录中可追加：

- `08_苹果中枢_阶段性法律分析报告_当前材料版.md`
- `09_交付前自检报告.md`
- `10_苹果审计组_阶段性审计报告.md`
- `11_进化系统_审计短板分析输入.md`
- `12_结构化补证任务清单.md`
- `13_外部依据与类案核验任务清单.md`

## 典型审计结论措辞

- “阶段性通过但禁止外部交付”
- “现有成果只能作为内部阶段性成果和补证指引”
- “不能认定为完整交付完成”

## 典型 blocked 原因

- 核心证据原件未入卷。
- 法条/司法解释仅内部初核，未完成外部权威原文核验。
- 类案未绑定案号、法院、裁判原文。
- 诉讼请求金额需根据已赔金额/第三方赔偿/工伤基金支付情况校准。
- 知识沉淀仍为草案，未经审计不得正式入库。
