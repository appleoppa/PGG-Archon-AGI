# 苹果中枢办案系统重建/清库/部门重映射工作流

适用场景：用户审核通过后，要求把苹果中枢办案系统的新 profile 架构全局落位，清除旧版配置、旧口径和旧办案记录，并从新架构重新开案。

## 关键口径

- `default` profile 是苹果中枢/main。
- `pgg-*` profile 是各部门。
- 文书机要部不再作为独立部门；文书起草、格式校对、终版润色并入业务主办部门内部自检。
- 原文书部门 profile 可按用户指令重映射为 `pgg-zhixing` 强制执行部。
- 强制执行部职责：执行立案、财产线索整理、查控申请、保全/查封/冻结/拍卖路径、执行异议与复议、终本复查、执行和解方案、执行风险评估。

## 安全顺序

1. **先备份**
   - 备份 `现行部门配置*`、workflow、template、standards、办案库、台账、审计队列、任务队列、相关 profile、相关 skill。
   - 备份目录建议：`workspace/存档/办案系统重建前备份/<timestamp>/`。
   - 写 `MANIFEST.json` 记录来源和目标。

2. **profile 重映射**
   - 如用户明确要求，把旧文书 profile 目录重命名为 `pgg-zhixing`。
   - 更新 `pgg-zhixing/SOUL.md` 为强制执行部身份。
   - 删除或重建该 profile 下继承自旧文书部门的局部 `skills/`，避免旧部门技能影子污染。

3. **正式入口落位**
   - 写入 `workspace/治理/部门配置/现行部门配置.md`，作为唯一现行入口。
   - 删除待审核草案，避免后续搜索命中旧口径。

4. **同步主流程与模板**
   - 更新 `workflows/办案流程详解.md`。
   - 更新 `workflows/法律办案_Hermes-Kanban链路.md`。
   - 更新 `templates/苹果中枢办案系统_指令模板.md`。
   - 将重复模板改成引用版或归档。
   - 更新 `standards/core/12_案件类型流程配置表.md`。

5. **同步 skills**
   - 至少 patch `apple-hub-orchestrator` 和 `agent-cms`。
   - 去除“14个部门”“独立文书机要部”“旧案件共享路径”“运行维护部作为部门”等现行口径冲突。

6. **清空办案记录**
   - 按用户明确授权再清空。
   - 重建空办案库：
     - `苹果中枢办案库/README.md`
     - `苹果中枢办案库/台账/案件台账_最新.md`
     - `苹果中枢办案库/台账/案件速记.md`
   - 台账写明当前有效案件 0，下一个编号 001。
   - 清空 `审计队列`、`任务队列` 的旧办案记录。
   - 删除旧顶层 `台账/`、旧 `案件共享/` 等不一致活跃入口。

7. **验证**
   - `pgg-wenshu` 不存在。
   - `pgg-zhixing` 存在且 SOUL 是强制执行部。
   - 办案库只剩 README 和台账目录。
   - 审计队列/任务队列为空初始化状态。
   - 核心活跃文件无旧口径匹配：`pgg-wenshu`、`PGG-WenShu`、`workspace/案件共享`、`案件共享/PGG`、`14个部门`、`运行维护部`。
   - 搜索时可排除 `存档/` 和 logs；存档命中是备份证据，不是现行口径。

## Pitfalls

- 不要把备份里的旧口径当成清理失败；报告时要区分“活跃配置”与“存档备份”。
- 删除旧办案记录是高影响操作；必须有用户明确授权且先备份。
- 待审核草案一旦正式落位，应删除或归档，否则未来搜索会误判冲突。
- profile 目录重命名后要检查局部 skills/logs，旧局部 skills 可能继续带旧路径和旧部门口径。
