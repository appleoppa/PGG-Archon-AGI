## 2. 两案带来的真实经验增量

### 已补足的真实经验

1. “启动办案程序”应直接触发正式办案流程，而不是中枢初筛。
2. 正式案号只能由案件管理中心生成，中枢不得预设正式编号。
3. 如果不能对外交付，必须生成内部办案流程报告，并且只把最终内部报告同步桌面。
4. 办案报告必须区分：流程已走完、证据 Gate Hold、内部预分析、可对外交付。
5. 金额类案件必须设置强制金额闭合核算，不闭合不得进入正式函件。
6. 法律依据和类案如果未实检，必须明确标注“初核方向 / 不得引用具体案例号”。
7. 部门调用失败或超时必须标注为流程异常，不得冒充部门真实完成。

### 尚未真正补足的短板

1. 实战结果尚未自动写入 PGG Archon 的 real capability metrics。
2. 两案暴露的错误尚未自动 append 到 failure sample library。
3. 两案结束后尚未自动生成标准化三问复盘进入 task retrospective ledger。
4. 办案系统和 PGG Archon 进化系统仍是人工同步，缺自动事件桥。
5. 证据门禁、金额门禁、编号门禁、部门调用门禁还未形成统一机器可读事件。

## 3. 进化计划

### P0：真实任务事件桥

目标：把办案流程产物转成 PGG Archon 可读事件。

范围：
- 扫描 `~/.hermes/workspace/苹果中枢办案库/` 下的新案件。
- 提取案件编号、阶段、门禁、部门节点、对外交付状态、内部报告路径。
- 生成只读事件：case_opened、department_flow_completed、evidence_gate_hold、internal_report_generated、delivery_blocked。

验收：
- 两案均能生成事件摘要。
- 不读取敏感原文，不外发，不改原案卷。

### P0：失败样本自动沉淀

目标：把两案暴露的流程错误写入 failure sample library。

首批样本：
1. 启动办案程序被降级为中枢初筛。
2. 中枢预设正式编号，违反案件管理中心编号权。
3. 部门调用超时后容易被误报为部门完成。
4. 金额不闭合仍推进联系函草稿。
5. 内部报告规则未在第一案开始前自动执行。

验收：
- 每条样本包含错误类型、触发场景、原因、正确做法、自动检测规则、下次拦截方式、证据 hash。
- 不保存客户敏感原文。

### P0：三问复盘自动生成

目标：每个案件阶段性结束后生成三问复盘。

三问字段：
- 发生了什么；
- 为什么发生；
- 下次怎么改。

两案初始复盘：
- PGG-MS：启动指令识别正确，但流程激活不完整；原因是中枢把“办案程序”降级为初筛；下次直接触发案件管理中心编号和部门并行。
- PGG-FW：材料处理和金额纠偏有效，但编号先错后补；原因是中枢越权预设编号；下次正式编号必须先走案件管理中心。

验收：
- `task_retrospective_status.retrospective_count` 从 0 增加到至少 2。

### P1：真实能力指标接入

目标：让 9 个真实能力指标从 UNKNOWN 变为有证据。

两案可喂养指标：
1. 事实查证与来源绑定：案卷文件、材料 OCR、报告路径。
2. 证据链完整性：P0 补证清单、证据 Gate Hold。
3. 法律依据核验：初核与未实检标注。
4. 工具执行与验收：文件生成、读取、路径验证。
5. 交付闭环：不能交付时内部报告生成。
6. 失败样本学习：流程错误沉淀。
7. 真实任务三问复盘：两案复盘。
8. 低风险自治候选：自动提出下一步补强，但不自动对外交付。
9. 多模型互补证据：后续可对复杂法律论证调用 GPT/Claude 复核，但办案中文法律可保留 DeepSeek 主处理。

### P1：办案门禁规则工程化

新增机器规则：
1. 用户文本包含“启动办案程序 / 开始办案 / 执行办案” → 必须触发正式流程。
2. 正式案号不匹配 `PGG-{类型}-{日期}-{序号}` 或非案件管理中心生成 → BLOCK。
3. 金额字段出现多个版本且差额未解释 → HOLD。
4. 证据原件未核验但输出“正式/终版/可对外交付” → BLOCK。
5. 部门调用失败却写“已完成”且无异常标注 → BLOCK。
6. 不能对外交付且无内部报告 → BLOCK。

## 4. 下一步执行顺序

## 5. 安全边界

- 不对外交付法律文书。
- 不自动写正式 gene/skill/memory。
